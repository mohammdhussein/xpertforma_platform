Auth

POST/api/auth/register/coach/

Public. Register a coach account. New coach accounts start in pending approval status.

Path Params

None

Request Body

{
  "first_name": "Coach",
  "last_name": "Name",
  "phone_number": "0501112222",
  "email": "coach@example.com",
  "password": "StrongPass123!",
  "certificate_image": "coach_certificates/certificate.png"
}

Response 201

{
  "id": "uuid",
  "email": "coach@example.com",
  "approval_status": "PENDING",
  "message": "Registration submitted. Please wait for admin approval."
}

Error Notes

  • 400 if first_name or last_name is missing
  • 400 if certificate_image is blank
  • 400 if the email is already registered

Expected Values

  • first_name, last_name, phone_number, email, password: required strings.
  • certificate_image: required uploaded image/file.
  • approval_status: always PENDING after success.
POST/api/auth/login/

Public. JWT login. Blocks coach accounts that are pending or rejected. Player logins return FIRST_LOGIN or COMPLETE in player_status.login_status.

Path Params

None

Request Body

{
  "email": "user@example.com",
  "password": "StrongPass123!"
}

Response 200

{
  "refresh": "jwt",
  "access": "jwt",
  "access_expires_at": "2026-04-02T18:30:00Z",
  "coach_status": {
    "register_status": "APPROVED"
  },
  "player_status": {
    "has_coach": true,
    "login_status": "COMPLETE"
  }
}

Response 400 (blocked coach)

{
  "coach_status": {
    "detail": "Coach account is pending approval.",
    "expected": ["APPROVED"],
    "register_status": "PENDING"
  }
}

Rejected coaches receive the same shape with detail = "Coach account has been rejected." and register_status = REJECTED.

Expected Values

  • email, password: required strings.
  • coach_status.register_status: PENDING, APPROVED, or REJECTED.
  • player_status.login_status: FIRST_LOGIN or COMPLETE.

POST/api/auth/logout/

Authenticated. Blacklists the current access token JTI in cache and revokes the provided refresh token.

Headers

Authorization: Bearer <access_token>

Request Body

{
  "refresh": "jwt"
}

Response 205

No content

Expected Values

  • Authorization: required Bearer <access_token> header.
  • refresh: required refresh JWT string. After logout it cannot be used at /api/auth/refresh/.
  • Response 205: no body.

AI Assistant

POST/api/ai/chat/

Authenticated player or approved coach. This endpoint is chatbot, app-data assistant, and agent in one endpoint: app-data questions such as latest sessions, attendance, progress, readiness, players, and plans are answered from permission-scoped Django backend context; normal non-data messages get a concise Groq chatbot answer; coach plan requests for one or more players trigger the server-side agent workflow. The mobile app sends a message to Django only; Django detects player names from the message inside the signed-in coach's roster, calls Groq server-side when needed, stores generated plan options as a server-side draft, renders safe mobile HTML, and returns native actions for Kotlin Compose buttons. The mobile app must not send player IDs, role, coach IDs, screen context, or HTML.

Request Body

{
  "message": "Suggest 3 plans for Ahmad",
  "response_format": "html"
}
{
  "message": "Suggest 3 training plans for Omar and Kareem",
  "response_format": "html"
}
{
  "message": "Suggest 3 training plans for Omar and Kareem Abo Salah both to start from the current day and to end in the tomorrow day",
  "response_format": "html"
}

Response 200

{
  "answer": "I can help with training, recovery, players, and plan creation.",
  "render_type": "html",
  "html": "<div style=\"...\">I can help with training...</div>",
  "cards": [],
  "actions": [],
  "suggested_questions": [
    "Suggest 3 training plans for one of my players",
    "How should I structure this week's training?",
    "What can you help me do as a coach?"
  ]
}
{
  "answer": "I drafted 3 training plan options for Ahmad Saleh.",
  "render_type": "html",
  "html": "<div style=\"...\">...safe plan cards with Start date, End date, Start, and End labels...</div>",
  "cards": [],
  "actions": [
    {
      "type": "select_plan_option",
      "label": "Use Speed Plan",
      "draft_id": "server-side-draft-uuid",
      "option_id": "option_1",
      "requires_confirmation": true
    }
  ],
  "suggested_questions": [
    "Suggest 3 recovery plans for Ahmad Saleh",
    "Suggest 3 speed plans for Ahmad Saleh",
    "How should I choose between these plans?"
  ]
}

Multiple Player Match

{
  "answer": "I found more than one matching player. Please choose the player before I draft plans.",
  "render_type": "html",
  "html": "<div style=\"...\">I found more than one matching player...</div>",
  "cards": [],
  "actions": [
    {
      "type": "select_player",
      "label": "Use Ahmad Saleh",
      "player_id": "player-uuid",
      "player": {
        "id": "player-uuid",
        "name": "Ahmad Saleh",
        "position": { "id": 12, "name": "Striker", "code": "ST" },
        "avatar_url": null
      },
      "requires_confirmation": false
    }
  ],
  "suggested_questions": [
    "Suggest 3 plans using the full player name",
    "What details can help identify the player?",
    "How should I choose a plan focus?"
  ]
}

Expected Values

  • message: required string, max 2000 characters.
  • response_format: optional, currently html.
  • Rejected request fields: selected_player_id, screen_context, user_id, coach_id, role, html, and player_id.
  • Normal chatbot answers return the same mobile response shape with empty actions and three suggested_questions. Normal answer HTML is rendered as a white card on a full-width white response surface, avoiding the light gray wrapper background. Session-status answers use a colored left border: green for COMPLETED, red for MISSED, blue for IN_PROGRESS, and gray for NOT_STARTED.
  • suggested_questions: always present on successful assistant responses and contains up to three follow-up prompts.
  • Plan requests can name one player or multiple players, for example Suggest 3 training plans for Omar and Kareem. If each name resolves to exactly one player owned by the signed-in coach, Django saves one draft for all matched players and confirmation assigns the created plan to all of them.
  • Plan requests can include simple natural date wording such as start from the current day, start today, and end in the tomorrow day. When detected, Django passes the requested range to Groq, stores it in the draft options, renders Start date and End date chips in the HTML, and validates that all preview sessions stay inside the range.
  • If a requested name matches multiple roster players, Django returns select_player actions instead of guessing.
  • For plan requests, Groq must return structured JSON with exactly three options. If Groq returns the wrong shape or option count, Django sends one repair prompt before returning an invalid-response error. Django converts valid JSON into sanitized HTML and never trusts model-generated HTML.
  • Plan HTML uses only div, p, h3, h4, span, strong, b, ul, li, and br, with inline style attributes only. JavaScript, forms, external CSS, external images, scripts, and buttons inside HTML are not allowed. Plan cards include explicit session Start and End time labels.
  • Assistant HTML uses the app theme colors: primary #1e6eeb, primary gradient #2f78ef, primary hover #1458c3, secondary #22c55e, orange #f59e0b, danger #ef4444, text #0f172a, muted #64748b, border #dbe5f0, background #f8fafc, soft background #fbfdff, and surface #ffffff.
  • actions[] are rendered by the mobile app as native Kotlin Compose controls.

Error Notes

HTTP 400   validation error, unsupported request field, invalid selected option, expired draft
HTTP 401   missing or invalid JWT
HTTP 403   player account, unapproved coach, draft owned by another coach, or player no longer owned by coach
HTTP 502   { "detail": "Groq returned an invalid plan response." }
HTTP 503   { "detail": "AI assistant is disabled" }
HTTP 503   { "detail": "Missing GROQ_API_KEY." }
HTTP 503   { "detail": "Groq service is unavailable." }
HTTP 503   { "detail": "Groq service is unavailable. Provider status: 429." }
HTTP 504   { "detail": "Groq request timed out." }

GET/api/coach/players/

Authenticated coach only. Returns the signed-in coach players list.

Query Params

  • q: search by player name, email, or position
  • tab: ALL, ACTIVE, INJURED, or NEEDS_ATTENTION

Response 200

{

  ]

Expected Values

  • q: optional search string.
  • tab: uppercase only: ALL, ACTIVE, INJURED, NEEDS_ATTENTION.
  • players[].state: stored player state, either ACTIVE or INJURED.
  • players[].needs_attention: server-computed boolean. Injury also counts as needs attention. Not-started sessions do not count as low progress until there is a final player outcome such as completed or missed.
  • players[].expected_return_date: YYYY-MM-DD or null.
  • players[]: player objects with position and avatar_url.

Error Responses

Errors for invalid tab include the accepted values in expected:

HTTP 400
{
  "detail": "Invalid tab. Use uppercase values.",
  "expected": ["ALL", "ACTIVE", "INJURED", "NEEDS_ATTENTION"]
}
PATCH/api/coach/players/<player_id>/state/

Authenticated coach only. Updates the stored player state for one of the signed-in coach's players.

Path Params

  • player_id: player user UUID

Request Body

{
  "state": "INJURED",
  "expected_return_date": "2026-05-01"
}

Response 200

{
  "player_id": "uuid",
  "state": "INJURED",
  "expected_return_date": "2026-05-01",
  "needs_attention": true
}

Expected Values

  • state: required uppercase token, either ACTIVE or INJURED.
  • expected_return_date: optional when state=INJURED, format YYYY-MM-DD.
  • Sending expected_return_date with state=ACTIVE returns 400.
  • Setting state=ACTIVE clears the stored expected return date.
  • needs_attention is recomputed after the state change. Not-started sessions do not count as low progress until there is a final player outcome.

Error Responses

HTTP 400
{
  "detail": "Invalid state. Use uppercase values.",
  "expected": ["ACTIVE", "INJURED"]
}
HTTP 400
{
  "detail": "Invalid expected_return_date format.",
  "expected": ["YYYY-MM-DD"]
}

Coach Plans

GET/api/coach/plans/

Authenticated coach only. Returns the signed-in coach's training sessions for an inclusive date range, grouped per day and per plan. Only days with at least one session are returned.

Query Params

  • start_date: optional, format YYYY-MM-DD. Defaults to today.
  • end_date: optional, format YYYY-MM-DD. Defaults to today.
  • Range must satisfy start_date <= end_date and span at most 31 days.

  "start_date": "2026-04-01",
  "end_date": "2026-04-07",
  "days": [
    {
      "date": "2026-04-01",
      "plans": [
        {
          "plan_id": "uuid",

Calculation Notes

  • sessions.planned: sessions from plans assigned through TrainingPlanPlayer within the selected range.
  • sessions.completed: session lifecycle is COMPLETED and the player has a SessionAttendance row for the session.
  • effort.percentage: completed intensity points divided by planned intensity points. Weights are LOW=1, MEDIUM=2, HIGH=3.

  • All percentages and scores are rounded to the nearest integer and clamped between 0 and 100.

  "mood": 5,
  "sore_zones": [],
  "readiness_score": 100,
  "created_at": "2026-04-16T06:00:00Z"
}

Expected Values

  • All fields are required; the full check-in is replaced.
  • sleep_hours: decimal 0–24.
  • sleep_quality: one of POOR, FAIR, GOOD, GREAT.
  • mood: integer 1–5.
  • sore_zones: array of zone strings (may be empty). Valid values: head_neck, shoulders, upper_back, lower_back, hip_groin, left_quad, right_quad, hamstring, knee, calf, ankle_foot.
  • readiness_score: server-recomputed integer 0–100.

Error Responses

  • 400: validation error (invalid field values).
  • 404: no check-in submitted for today.

Invalid sleep_quality includes the accepted values in expected:

HTTP 400
{
  "detail": "Invalid sleep_quality.",
  "expected": ["POOR", "FAIR", "GOOD", "GREAT"]
}

Invalid sore_zones includes the accepted zones in expected:

HTTP 400
{
  "detail": "Invalid sore_zones value.",
  "expected": ["head_neck", "shoulders", "upper_back", "lower_back", "hip_groin", "left_quad", "right_quad", "hamstring", "knee", "calf", "ankle_foot"]
}

GET/api/user-info/

Authenticated. Returns the signed-in user profile summary and role-specific data.

Response 200: Player Example

{
  "id": "uuid",
  "name": "User Name",
  "first_name": "User",
  "last_name": "Name",
  "email": "user@example.com",
  "role": "Player",
  "player": {
    "position": {
      "id": 12,
      "name": "Striker",
      "code": "ST"
    },
    "team_id": "uuid",
    "height_cm": 182.0,
    "weight_kg": 77.0,
    "avatar_url": "/media/player_avatars/avatar.png"
  }
}

Response 200: Coach Example

{
  "id": "uuid",
  "name": "Coach Name",
  "first_name": "Coach",
  "last_name": "Name",
  "email": "coach@example.com",
  "role": "Coach",
  "coach": {
    "approval_status": "APPROVED",
    "certificate_image": "/media/coach_certificates/certificate.png",
    "approved_at": "2026-04-02T14:00:00Z",
    "rejection_reason": ""
  }
}

Role Notes

  • If role is Player, the coach object is omitted.
  • If role is Coach, the player object is omitted.

Expected Values

  • role: primary role string such as Player or Coach.
  • first_name and last_name: split name fields; either can be an empty string.
  • player.position: object with id, name, code; avatar_url is always present.
  • coach.approval_status: PENDING, APPROVED, or REJECTED.
